Current State

The Pain Points We're Solving

Manual Google Drive folders per student — no centralised tracking, no automation
Document verification (VSC, CPR, CPI etc.) tracked manually with no alerts
No formal hours tracking — relies on task output as an indirect proxy
WhatsApp coordination — informal, not auditable, easy to miss messages
No gender/safety pairing rules enforced in the system — all manual judgment
Volunteer feedback as freeform Google Docs — can't report or analyse
No proximity-based matching — coordinators assign manually without distance data
V4S doesn't natively support school-mediated student intake — requires custom build
Post-placement transitions tracked manually — no system record of who stays on
Access revocation relies on memory — no automated off-boarding trigger
04 / 12
Key Discovery

Two Classes of Volunteers

PRIMARY — Majority

👩‍🎓 Student Volunteers

  • Source: Colleges / Universities (formal placement)
  • Entry: School submits → H&M reviews → interview → docs
  • Programmes: PSW, CDSW, DSW, BSW, MSW, AMH
  • Basic: in-home tasks (laundry, meals, errands)
  • Advanced: emotional support, supervision, referrals
  • Work: 30% in-person, 70% virtual
  • Requires: 5 compliance documents from school
  • V4S fit: Custom record type + Student Placement object
SECONDARY — Low Volume

🌍 General Volunteers

  • Source: Charity Village, website, referrals
  • Entry: Web form → coordinator interview → onboard
  • Roles: Event support, outreach, respite
  • Documents: Vary by role (VSC for respite, lighter for events)
  • Volume: ~3 to date (low but growing)
  • Many are ex-students who stayed after placement
  • V4S fit: Standard V4S web-to-lead flow works well
🏗️ Architecture Implication: Both types use the Salesforce Contact object with different Record Types. Students additionally link to a custom Student Placement object that tracks all placement-specific data.

Phase 4 Deep Dive

Volunteer Matching Safety Rules

📍
Proximity Matching
Volunteer addresses are geocoded. The system sorts potential volunteers nearest-first to the caregiver's location. Volunteers beyond a configurable radius are excluded unless manually added. Student volunteers included when they meet distance rules.
⚖️
Gender Safety Pairing
If a female caregiver or children are present in the home, the system blocks a 2-male pairing. Valid pairings: Male + Female, or Female + Female. Coordinator can override with a mandatory justification note stored for audit.
🚗
Driving Capability
For caregivers beyond transit distance, the system requires at least 1 volunteer with Drives = Yes. Assigning 2 non-driving volunteers to a far-distance case is blocked with a clear error. Drives field stored on every volunteer Contact.
🛠️
Skills Matching
Volunteers tagged with skills: Meal Prep, Laundry, Housekeeping, Mental Health Support, Outreach, Admin. System filters by skills required for the caregiver's support request. Partial match shows with a warning — coordinator decides.
⭐
Top 3 Recommendations
After filters applied, system shows top 3 recommended volunteer pairings with a score. Pairings that violate any mandatory rule are flagged with warning icons — not hidden.
🔁
Cancellation Handling
Caregiver cancels: Case rescheduled, shift cancelled, volunteers notified. Volunteer cancels: System suggests replacement from available pool. All cancellations logged on the Case for audit and reporting.

Quality Assurance

The Gated Approach — How It Works

Nothing moves to the next phase until the current gate is confirmed. This protects Phase 1 (already live) and ensures every build step is validated before we layer on complexity.

1
Built and tested in sandbox by developer
2
BA tests against all Jira acceptance criteria
3
Client (Abigail) reviews and confirms — user acceptance testing
4
Regression check: Phase 1 still works (no broken cases)
5
Moved to production — gate closed ✅
Roles & Responsibilities
BA Team (Taiwo, BC)
Story refinement, UAT, client communication, gate sign-off
Developer Team (Ital + team)
Flow builds, custom objects, validation rules, automation
Client — Abigail / Hearts & Mind
UAT sign-off at each gate, email template content, process verification
Project Manager
Gate tracking, sprint planning, risk management
